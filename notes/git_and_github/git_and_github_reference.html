Git and GitHub

Git is version control. GitHub is where that version control lives online. The bootcamp treats Git as mandatory from day one. This reference covers the seven commands you need to know before you arrive, what each one does in plain English, and the mental model that makes all of them make sense.

NOTE 01

The Mental Model

Plain English - what Git actually is

Git tracks changes to files over time. Every time you tell it to, it takes a snapshot of your project. You can go back to any snapshot. You can share snapshots with other people. That is the whole thing.

Think of it as a save-game system for code. Every commit is a save point. If something breaks, you load an earlier save.

Git does not back up automatically. You tell it when to save.
Three Zones - where your work lives
ZoneWhat it contains
Working directoryThe files you are editing right now. Git watches but does not save yet.
Staging areaFiles you have marked as ready to be saved in the next snapshot.
RepositoryThe permanent record of every snapshot you have committed.
git add moves work from zone 1 to zone 2. git commit moves it from zone 2 to zone 3.
Rule - commit often, commit small

A commit should represent one logical change, not a day's work.

Good commit message: Add filtering to the sales notebook

Bad commit message: stuff or changes or final final v3

The message is for future you. Be specific.

You will read your own commit messages in six months. Write them accordingly.
NOTE 02

One-Time Setup

Configure Git - run these once, ever

Before using Git for the first time, tell it who you are. These two commands set your name and email on every commit you make. Run them in the terminal once. They persist across all projects.

git config --global user.name "Your Name"Sets your name on all commits. Replace with your actual name.
git config --global user.email "you@email.com"Sets your email on all commits. Use the same email as your GitHub account.
Run these in the terminal before anything else. Once only.
NOTE 03

The Seven Commands

Scope - what you need before the bootcamp

These seven commands cover everything you will need in the first weeks of the bootcamp. Learn them in order. Each one builds on the last.

Seven commands is enough to work professionally with Git.
git init - start tracking a project

Run this once inside a project folder to tell Git to start tracking it. It creates a hidden .git folder that stores all the version history. You never touch the .git folder directly.

One project, one git init, done once.
git status - what is happening right now

Shows you which files have changed, which are staged, and which are untracked. Run this constantly. It costs nothing and tells you exactly where you are.

When in doubt, git status. Always.
git add - stage a file for saving

Marks a file as ready to be included in the next commit. git add filename.py stages one file. git add . stages everything that has changed in the current folder. Staging is reversible. Committing is permanent.

You are choosing what goes into this snapshot. Choose deliberately.
git commit - take the snapshot

Saves everything in the staging area as a permanent snapshot with a message. The -m flag lets you write the message inline. git commit -m "your message here" Every commit gets a unique ID. You can always return to it.

This is the save point. The message is the label on the save.
git log - see the history

Shows the list of all commits made so far, newest first. Each entry shows the commit ID, author, date, and message. Press q to exit the log view.

git log tells you where you have been. git status tells you where you are.
git clone - copy a remote repository to your machine

Downloads a complete copy of a repository from GitHub to your local machine, including all its history.

So if you wanted to clone this Git repo, for instance, you would enter:

git clone https://github.com/TidyWeb/learn-python-data.git

The bootcamp will likely give you a repo to clone on day one.

Clone once. Work locally. Push back when ready.
git push - send your commits to GitHub

Uploads your local commits to the remote repository on GitHub. git push origin main origin is the name Git gives to the remote by default. main is the name of the primary branch.

Nothing reaches GitHub until you push. Local commits stay local.
NOTE 04

The Typical Workflow

The Loop - what you do every session
StepCommand
Check what has changedgit status
Stage the files you want to savegit add filename or git add .
Commit with a clear messagegit commit -m "describe what changed"
Send to GitHubgit push origin main
Status, add, commit, push. Repeat every session.
Rule - push before you finish for the day

A commit that lives only on your local machine is at risk. Push to GitHub at the end of every working session. GitHub is the backup. Your local machine is not.

If your laptop died tonight, would your work survive? Push and it would.
NOTE 05

GitHub vs Git

Plain English - two different things

Git is the version control system. It runs on your machine. It has no interface.

GitHub is a website that hosts Git repositories online. It adds collaboration, visibility, and backup to what Git does locally.

You can use Git without GitHub. You cannot use GitHub without Git.

Git is the tool. GitHub is where the tool's output lives online.
What GitHub Adds - beyond local version control
FeatureWhat it means in practice
Remote backupYour commits are safe even if your machine fails.
PortfolioYour public repositories are visible to employers.
CollaborationOthers can clone, contribute, and review your work.
Pull requestsA formal way to propose changes before they are merged.
The bootcamp will use GitHub as the submission and collaboration platform.
