SQL Readiness Hand-off

Python prepares the table. SQL asks questions of it. This short chapter explains the decisions that make that hand-off deliberate, then introduces the small browser adaptation used by the next bridge.

What “ready for SQL” means

A table is ready when you can explain what one row means, use names that are easy to query, trust the main value types, account for missing values, and distinguish accidental duplicates from legitimate repeated records.

  1. One row has one clear meaning.
  2. Column names are lowercase, consistent, and easy to type.
  3. Dates behave like dates and measurements behave like numbers.
  4. Missing values and duplicates have deliberate decisions behind them.
  5. The final table is named prepared_df.

The preparation decisions

1. One-row meaning

Start with the table's grain: what does one row represent? A SQL query is easier to trust when you know whether a row is one workout, one order, one person, or one event.

df.shape

shape reports the row and column count. It helps you notice whether a cleaning decision unexpectedly changed the table's size.

2. Useful column names

Names such as activity_date, duration_minutes, and max_pulse are easier to type and read than names containing spaces or punctuation.

df = df.rename(columns={"Date": "activity_date"})

rename changes labels; it does not change the values inside the table.

3. Deliberate types

Inspect the types before SQL receives the table. Convert date-looking text to dates and number-looking text to numbers only after checking that the conversion makes sense.

df["activity_date"] = pd.to_datetime(df["activity_date"])

dtypes is the quick inspection that shows what pandas is using for each column.

4. Missing values and duplicates

Inspect first. Then decide whether to fill, remove, or preserve a missing value. Remove a duplicate only when you know it is accidental; repeated events can be real data.

df.isna().sum()
df.duplicated().sum()

The hand-off name: prepared_df

prepared_df is not a special pandas word. It is the agreed contract between Python preparation and the bridge. The bridge stays locked until it finds a valid prepared table under that exact name.

Before opening the bridge, print the table's shape, column names, types, missing-value counts, and first rows. Then explain why your decisions make the table ready for the next step.

Why the browser uses StringIO(csv_text)

A local notebook can read a file path such as pd.read_csv(dataset_path). The browser bridge holds the CSV characters in a variable called csv_text.

StringIO wraps that text so read_csv can read it as file-like input. It does not clean or reshape the data.

prepared_df = pd.read_csv(StringIO(csv_text))
