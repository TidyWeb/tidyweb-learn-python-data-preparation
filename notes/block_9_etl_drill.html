Block 9 — ETL Drill

Block 9 is where the learner rehearses the full ETL rhythm in a tighter, more operational way. The task is not open-ended exploration. The task is to load a raw dataset, clean it, transform it into a better working form, save the cleaned result, and explain what was changed and why. The drill is run twice, and the second run is timed.

NOTE 01

What This Block Is

Plain English - from analysis to preparation

ETL stands for Extract, Transform, Load. In plain English, it means taking in raw data, improving it into a usable form, and producing a cleaned output that someone can work with next.

This block is not about finding insights. It is about getting data into a usable state. The learner is being asked to work like the person who prepares the table before analysis begins properly.

That means the quality of the process matters: preserving the original, naming the cleaned output clearly, and being able to explain each change made to the data.

Block 9 is about preparing trustworthy data, not interpreting it.
Rule - always keep the raw file intact

The original input file must never be overwritten. The learner works from a copy in memory, then exports a new cleaned CSV with a clear name that signals it is the processed version.

Raw in. Clean out. Never destroy the source.
NOTE 02

The ETL Pipeline

The three stages - extract, transform, load
Stage What happens here
Extract Read the raw CSV, inspect the columns, inspect missingness, inspect types, and establish what the table appears to contain.
Transform Clean missing values, fix types where needed, standardise values, remove or isolate broken rows, and create any columns needed to make the data usable.
Load Write the cleaned dataframe back out as a new CSV so the result can be reused later without repeating every cleaning step.
Extract is inspection. Transform is repair. Load is saving the usable result.
Typical moves - pipeline skeleton
df = pd.read_csv("raw_file.csv")
df.info()
df.isnull().sum()

df_clean = df.copy()

# clean or transform here

df_clean.to_csv("clean_file.csv", index=False)
NOTE 03

Required Outputs

What the learner must produce - not just code

Each ETL run should end with three things:

— a cleaned output CSV saved with a clear name

— a notebook record showing the cleaning and transformation steps

— a short explanation of what changed and why those changes were justified

What the explanation should cover - minimum reporting standard
What was wrong? Name the problem clearly: missing values, bad types, inconsistent labels, impossible values, duplicate rows, or unusable formatting.
What did you do? State the specific action: fill, drop, convert, rename, split, combine, recode, or export.
Why that move? Explain why the chosen fix makes sense for this table rather than just saying it worked.
What changed? Show the result in concrete terms: row count, null count, data type, or a visible before-and-after comparison.
The learner has to justify the cleaning, not just perform it.
NOTE 04

The Timed Second Run

Drill structure - two passes

The first ETL run is a steady untimed pass. The learner can think, inspect, and explain carefully. The second ETL run uses a different dataset and is timed.

The timer is part of the design. It forces the learner to move from recognition into fluency. When the time expires, the learner stops whether or not the job is fully complete.

Rule - stop when the timer ends

The point of the timed run is not perfect completion. The point is to reveal which steps are fluent already and which steps still slow the learner down. Partial completion is still useful evidence.

The timer is measuring fluency, not punishing incompletion.
