Git and GitHub Exercise Walkthrough

This walkthrough covers the standard sequence for putting a project under local Git, connecting it to GitHub, and then publishing a GitHub Pages site. Replace the placeholder path, email, and repository name with the real values for your own project.

PART 01

One-Time Setup

Before any project - - machine-level Git identity

These commands are not project-specific. They are machine-wide Git setup.

Run once on the machine, not once per repo.
Step 1 - - set your name
git config --global user.name "Your Name"
What it does - - commit identity

Registers the human-readable author name that will appear on every commit you make on this machine.

This does not have to match your GitHub username. Your commit author name can be your real name even if your GitHub account uses a different handle.

Use the name you want shown as the author on your commits. No output usually means it worked.
Step 2 - - set your email
git config --global user.email "your-github-email@example.com"
What it does - - commit identity

Registers your email on every commit. Use the same one as your GitHub account.

Again, no output usually means success.
PART 02

Local Git Setup

Project path - - replace with your project folder
cd "/path/to/your/project-folder"
Start Git - - local repo only
git init
Check state - - current branch and files
git status
Rename branch - - modern primary branch name
git branch -m main
Beginner note - - what the -m means

The word rename is not typed. Here, -m is the Git option doing the move/rename.

-m is specific to this Git command, not a universal terminal rename command.
Create ignore file - - normal terminal command
touch .gitignore
Command distinction - - terminal vs Git

touch is a normal terminal/Linux command for creating an empty file. It is not a Git command. Git then reads the file you created.

Some commands in a Git workflow belong to the shell. Some belong to Git itself.
Ignore rules - - file contents
.ipynb_checkpoints/
__pycache__/
*.py[cod]
.pytest_cache/
Stage everything - - first snapshot prep
git add .
First commit - - local save point
git commit -m "Initial commit"
Boundary - - what has happened so far

At this point, version control exists and works locally. Nothing has gone to GitHub yet.

Git first. GitHub later.
PART 03

Connect to GitHub

Create the GitHub repo first - - empty repo only

Create the remote repository on GitHub with no README, no .gitignore, and no license at creation time.

Choose a repository name that fits your project and GitHub naming conventions.

Keep the GitHub repo empty so the first push stays simple.
Add the remote - - connect local to GitHub
git remote add origin https://github.com/your-username/your-repository-name.git
Verify the remote - - fetch and push URLs
git remote -v
First push - - upload the local history
git push -u origin main
What the -u means - - future pushes become shorter

The -u flag tells Git to remember that local main is partnered with remote origin/main.

After this, later pushes can usually be just git push.
PART 04

GitHub Pages

Website publishing - - GitHub-side setup

Once the repo was on GitHub, Pages was configured on the website, not in the terminal.

Settings → Pages → Deploy from a branch → main → / (root).

Because index.html is in the repo root, Pages can publish directly from the root folder.
PART 05

Daily Workflow

The loop - - what happens after the first setup
git status
git add .
git commit -m "describe what changed"
git push
Meaning - - status, stage, save, upload

This is the ongoing maintenance loop once the repo already exists locally and already knows its GitHub partner.

Check, choose, save, send.
